Frequently Asked Questions

What is a public API?

A public API (Application Programming Interface) is a structured interface that allows external developers to request data or functionality from an online service. Think of it as a standardized way for software programs to communicate with each other over the internet.

Public APIs make their endpoints accessible to anyone — not just internal engineering teams — typically at no cost or through a generous free tier. This means you can build an application that displays real-time weather data, live stock prices, movie information, sports scores, or geographic details without having to collect or maintain that data yourself. You simply make a structured request to the API, and it returns a response, usually in JSON format, that you can use in your own application.

For example, instead of building a weather data collection system — which would require satellites, sensors, and meteorologists — you can call a weather API with a city name and receive a structured response containing temperature, humidity, wind speed, and a five-day forecast in milliseconds. Public APIs are the infrastructure layer that makes modern app development efficient and accessible for individual developers and small teams.

How do I get started using a public API? A step-by-step guide for beginners.

Getting started with a public API is more straightforward than it may seem. Here is a beginner-friendly step-by-step process:

Step 1 — Find a suitable API: Use Public API Finder to identify an API that matches your project's data needs. If you are just learning, start with a "No Auth" API — these require no registration and let you make requests immediately.

Step 2 — Read the documentation: Every API has a documentation page that explains what endpoints are available, what parameters each endpoint accepts, and what the response looks like. Spend at least 10–15 minutes reading the docs before writing any code. Good documentation will include example requests and responses that you can copy and test immediately.

Step 3 — Register and get an API key (if required): If the API requires an API key, sign up on the API provider's website. This usually involves entering an email address, verifying it, and then finding the API key in your account dashboard. Copy and store this key securely — it identifies your application and tracks your usage against rate limits.

Step 4 — Test the API manually: Before integrating the API into your code, test it using a browser (for simple GET requests) or a tool like Postman, Insomnia, or the curl command in your terminal. Making a manual test request first helps you verify the API is working and understand the response structure before writing any integration code.

Step 5 — Integrate into your application: Use your language's HTTP library (fetch in JavaScript, requests in Python, etc.) to call the API from your application code. Parse the JSON response and use the data in your UI or application logic. Handle errors gracefully — APIs can occasionally be unavailable or return unexpected responses, and your application should degrade gracefully when that happens.

What does "No Auth" mean, and when should I choose it?

"No Auth" means the API does not require any form of authentication to access. You can call the API endpoint directly — no API key, no OAuth token, no account registration of any kind. You simply send a request to the URL and receive data in response.

No Auth APIs are ideal in several situations. If you are a complete beginner learning how APIs work, starting with a No Auth API removes all setup friction and lets you focus on understanding the request-response cycle, JSON parsing, and data handling. If you are building a quick prototype or hackathon demo that needs to go live in hours, No Auth APIs eliminate the 15–30 minutes typically required to register, verify an email, and configure API key authentication. They are also perfect for static website integrations and situations where you want to call an API directly from client-side JavaScript without exposing an API key in your frontend code.

The trade-off is that No Auth APIs tend to be more restricted — they typically serve public, non-sensitive data (trivia questions, country information, fun facts, etc.) and may have lower rate limits than API-key-based services. For production applications that need reliable, high-volume access, you will usually want to move to an API key or OAuth-based service. But for learning and prototyping, No Auth APIs are an excellent starting point.

What is CORS and why does it matter for web developers?

CORS stands for Cross-Origin Resource Sharing. It is a security mechanism built into all modern web browsers that restricts JavaScript running on one domain from making HTTP requests to a different domain, unless that other domain explicitly allows it.

Here is why this matters in practice: if your web application is hosted at myapp.com and it tries to call an API at api.someservice.com, the browser will first send a "preflight" request asking the API server whether it permits cross-origin requests from myapp.com. If the API server's response includes the appropriate CORS headers (specifically the Access-Control-Allow-Origin header), the browser allows the request. If not, the browser blocks the request entirely and returns a CORS error — even if the API itself is perfectly functional and publicly accessible.

This is why our directory displays a "CORS" badge for each API. If an API's CORS value is "Yes," you can call it directly from client-side JavaScript in a browser without any workaround. If it is "No" or unknown, attempting to call it from frontend JavaScript will likely fail with a CORS error. In that case, you would need to route the request through a backend server (which is not subject to browser CORS restrictions), use a CORS proxy, or choose a different API. For frontend developers and students building browser-based applications without a backend, filtering the directory to CORS-enabled APIs is one of the most practical time-savers we offer.

How do I test an API before integrating it into my project?

Testing an API before writing integration code is one of the most important habits you can develop as a developer. Here are the most practical methods, from simplest to most capable:

Browser URL bar (for GET requests, No Auth APIs): The simplest test. If an API's documentation shows a GET endpoint — for example, https://api.example.com/data?city=London — you can paste that URL directly into your browser's address bar and see the raw JSON response. This works only for GET requests and only for APIs that require no authentication. It is instant and requires no tools.

curl (command line): For APIs that require an API key or have more complex request structures, curl is a powerful command-line tool available on all major operating systems. You can set custom headers, send POST request bodies, and see the full response including HTTP status codes and response headers. The API documentation will usually include example curl commands you can copy and run directly.

Postman or Insomnia (GUI tools): These are dedicated API testing applications with graphical interfaces. They let you save requests, organize them into collections, set environment variables (ideal for API keys), inspect response headers, and test different request configurations. Postman has a free tier that is more than sufficient for personal API testing, and it is the industry standard for API development and testing.

Whichever method you use, the key things to verify before integration are: the API returns the data structure you expect, the response time is acceptable for your use case, the authentication method works correctly, and the rate limits will accommodate your expected usage volume.

Are all the APIs in the directory truly free to use?

Our directory focuses on public APIs that offer free access, but it is important to understand what "free" means in the context of API services, because the specifics vary significantly.

Truly free, no limits: Some APIs — particularly those marked "No Auth" — are completely free with no registration, no usage limits, and no cost regardless of how many requests you make. These are typically smaller projects, open data sources, or APIs maintained by enthusiasts and organizations that want to make data freely available.

Free tier with rate limits: Many popular APIs offer a free tier that allows a certain number of requests per day or per month. For example, a weather API might allow 1,000 API calls per day for free, after which you need to upgrade to a paid plan. For personal projects, student work, and most side projects, these free tiers are more than sufficient. However, if your application grows significantly, you may eventually need to pay for higher rate limits.

Free with API key registration: Some APIs require you to register for a free API key to access even the free tier. Registration is typically free and takes only a few minutes, but it means you cannot use the API without first creating an account with the provider. We mark these APIs with the "apiKey" authentication badge in our directory.

We always recommend checking the current pricing page of any API you plan to use for a production application, as pricing and free tier limits can change without notice. Our directory data reflects the APIs' availability at the time of curation, but API terms evolve over time.

How often is the API directory updated?

Our API directory is maintained as a curated static dataset derived from the public-apis GitHub repository, which is itself actively maintained by a large open-source community. We perform periodic reviews and updates to our dataset to reflect changes in API availability, authentication requirements, CORS support, and the addition of new APIs that meet our inclusion criteria.

In practice, the core dataset is quite stable — the vast majority of well-established public APIs in our directory have been available for years and change infrequently. The most common changes we track are: APIs that shut down or move to paid-only models, changes in authentication requirements (for example, an API that previously required no auth adding an API key requirement), and the addition of new APIs that become popular in the developer community.

If you discover that a listed API has changed significantly — its endpoint has moved, it now requires authentication, it has been discontinued, or it has updated its CORS policy — please let us know at taeshinkim11@gmail.com. Community-reported updates are the fastest way for us to maintain accuracy at scale, and we appreciate every report we receive.

What is rate limiting and how should I handle it?

Rate limiting is a restriction placed by an API provider on how many requests you can make within a given time window. For example, an API might allow 60 requests per minute, 1,000 per day, or 100,000 per month on its free tier. When you exceed the limit, the API returns an error response — typically HTTP 429 (Too Many Requests) — instead of the expected data.

Rate limits exist for several good reasons. They protect the API provider's infrastructure from being overwhelmed by a single abusive user, they allow providers to offer free tiers sustainably without those free users impacting paid customers, and they encourage developers to build efficient applications that cache responses rather than making redundant requests.

Handling rate limits gracefully is an important part of building robust API integrations. Key strategies include: caching API responses so you are not re-requesting the same data unnecessarily; implementing exponential backoff (waiting progressively longer before retrying after a 429 error); keeping track of how many requests you have made and slowing down proactively as you approach your limit; and, for user-facing applications, showing a helpful message to users rather than a raw error when rate limits are hit.

Before committing to an API for a production application, always evaluate whether the free tier's rate limits are sufficient for your expected traffic. It is worth estimating your request volume at both your current user base and at 10x growth to make sure you will not hit a wall at an inconvenient moment. If the free tier is insufficient, many APIs offer affordable paid plans that dramatically increase rate limits — the documentation's pricing page will give you the numbers you need to plan ahead.
